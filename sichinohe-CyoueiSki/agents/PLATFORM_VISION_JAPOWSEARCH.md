# プラットフォーム展望 — japowserch.com と全国ゲレンデ連動

七戸町営スキー場の単体サイトは「現場の顔」でありつつ、別エージェントで運用する **japowserch.com** をハブに、日本全国のゲレンデ情報を**収集・正規化・解析**し、ランキングやマップUIと接続するインフラを目指す。

> ドメイン表記ゆれ（例: `japowsearch`）がある場合はDNS・ブランドの単一ソースを決めて統一する。

---

## レイヤー構成（論理）

```
各ゲレンデ公式サイト／CMS（テナント）
        │  公開フィード or 管理画面入力 or スクレイピング（最終手段）
        ▼
収集ジョブ（並列ワーカー） ──► 正規化ストア（時系列 + 最新スナップショット）
        │
        ├──► japowserch.com（マップ・ランキング・検索・比較）
        └──► 各テナントサイト埋め込み（任意） … 「今日」ウィジェット同期表示
```

- **単体サイト（例: 七戸）**: ブランド・ローカルストーリー・長尾LP・コンプライアンスの本体。
- **ハブ（japowserch.com）**: 横断比較・ランキング・地図・通知の本体。
- **連動の価値**: 運営は一度入力／連携すれば、ハブと自サイト双方で一貫した「今日の事实」を出せる（二重メンテ負荷を減らす設計が鍵）。

---

## マップのポップアップに載せる情報（一目）

ポップアップは**読み切り3〜5秒**を前提に、次の**正規化フィールド**から優先順位付きで最大N項目表示する（詳細は `DATA_MODEL_NORMALIZED_RESORT.md`）。

| 優先 | 項目例 |
|------|--------|
| P0 | 営業状態（営業中／時間外／不明）、最終更新時刻 |
| P0 | 積雪代表値・24h降雪（データ由来をアイコンで） |
| P0 | リフト運行サマリー（本数／遅延・停止の有無） |
| P1 | 本日の営業時間帯 |
| P1 | 主要レストラン・売店の稼働サマリー |
| P2 | 託児・キッズルーム稼働 |
| P2 | スクール当日受付（受付中／満員／終了） |

「不明」は悪ではなく**信頼のため明示**する（推測で緑点灯しない）。

## ランキング・解析

- **入力**: 正規化された時系列（降雪イベント、運行率、更新遅延、混雑プロキシ※）など。
- **出力例**: エリア別「降雪地」「運行安定」「コスパ」「ファミリー友好」など複数軸；単一指標ランキングは誤解を招くため**軸と定義をページ上で開示**する。
- **公平性**: データ欠損リゾートをどう扱うか（除外／ペナルティ／「データ協力募集中」）をポリシー化。

※混雑はウェブだけでは正確に取れないことが多いため、将来は提携データや推定モデルを任意レイヤーとして分離。

## 運営更新（販売後・自社運用）

現場・貴社どちらが更新しても同じデータへ書き込む **低難易度の更新コンソール** と、将来の **LINE 連携（Messaging API → 正規化ストア）** は、`ADMIN_CONSOLE_AND_LINE_UPDATES.md` およびエージェント12で仕様化する。ハブ・単体サイト・マップポップアップはいずれもこの単一ソースから読む設計が理想。

## 収益モデル（ユーザー案の整理）

| 要素 | 説明 |
|------|------|
| 初期販売（例: 10万円） | テーマ／CMS連携／ウィジェット初期設定／オンボーディングをパッケージ化する場合の**スコープ境界**を見積もり前提で定義（無制限カスタムは不可に）。 |
| サブスク（更新料） | コネクタ維持、収集ジョブ、ダッシュボード、SLA（更新遅延閾値）、優先サポートの対価。 |
| アフィリエイト | リフト割引・トラベルサイトは**広告・ASPの規約遵守**と**ステマ規制・表示義務**を満たす（別エージェントでチェックリスト化）。 |

法務・表示は `10-monetization-packaging-affiliate.prompt.md` と `07-stakeholder-compliance.prompt.md` を連動させる。

## 七戸サイトとの整合

- 七戸は **CANONICAL_SOURCE** の実験場として、フィールド定義・更新UX・POV運用を先に固める。
- ハブ側マップポップアップは、七戸から供給される正規化JSONと**同一スキーマ**で描画できることを最初の受け入れ基準にする。
